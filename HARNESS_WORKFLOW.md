# Harness Engineering Workflow

このドキュメントは、ハーネスエンジニアリングの原則とワークフローを詳しく説明します。

## コア原則

### 手書きコード禁止

このプロジェクトでは、すべてのコードはエージェント（Claude Code）によって生成されます。
人間のエンジニアは以下に専念します：

- 環境の設計
- 意図の明確化
- フィードバックループの構築
- 計画のレビューと承認
- 成果の検証

### コード実装前に計画を必ずレビューして承認する

**最も重要な原則**: 計画と実行の分離により、無駄な作業を防ぎ、アーキテクチャ決定の管理を維持します。

## 3段階のワークフロー

### 第1段階：リサーチ

エージェントに対して深い読み込みを指示し、関連するコードベースを徹底的に理解させます。

#### プロンプト例
```
このフォルダ [path/to/folder] を深く読み込み、詳細なレポートを
docs/exec-plans/active/[task-name]-research.md に記述してください。
```

#### リサーチドキュメントに含めるべき内容
- 現在のコードベース構造
- 関連する既存機能
- 使用されている技術スタック
- 既存のパターンとコンベンション
- 潜在的な課題や制約
- 参考になる既存コード

**重要**: findings は必ずマークダウンファイルに**永続的に記録**します。
コンテキストウィンドウの圧縮後も保持されます。

### 第2段階：計画作成

詳細な実装計画を作成させます。

#### プロンプト例
```
[task description] の実装計画を docs/exec-plans/active/[task-name]-plan.md
に作成してください。既存のコードを参考にし、ARCHITECTURE.md の原則に従ってください。
```

#### 計画ドキュメントに含めるべき内容

1. **目標と成功基準**
   - 何を達成するのか
   - どうやって成功を測定するのか

2. **アーキテクチャ上の変更**
   - 新規作成するファイル/モジュール
   - 変更する既存ファイル
   - 依存関係の変更

3. **実装ステップ**
   - タスクを小さな段階に分解
   - 各ステップの具体的な作業内容
   - 完了条件

4. **テスト戦略**
   - 単体テスト
   - 統合テスト
   - E2Eテスト（必要に応じて）

5. **既知のリスクと制約**
   - 技術的な課題
   - 外部依存関係
   - パフォーマンスへの影響

6. **代替案（検討した場合）**
   - なぜこのアプローチを選んだのか

### 第3段階：アノテーションサイクル

生成された計画文書に**インラインコメントを直接追加**します。
これを1～6回繰り返し、計画を洗練させます。

#### アノテーションの例

```markdown
## Step 3: Implement SensorRepository

Create `src/domains/sensor/repository/SensorRepository.ts` with methods:
- `fetchLatestData()` - Get latest sensor readings
<!-- FEEDBACK: この名前は曖昧です。`fetchLatestSensorReadings()` に変更してください -->

- Use axios for HTTP calls
<!-- FEEDBACK: axiosではなくfetchを使用してください。依存関係を最小限にするため -->

- Cache results for 1 minute
<!-- APPROVED: これでOKです -->

- Return raw API response
<!-- FEEDBACK: 生のレスポンスではなく、型付きデータを返してください。
境界でZodを使ってバリデーションしてください -->
```

#### アノテーションの種類

- `<!-- FEEDBACK: ... -->` - 修正が必要
- `<!-- QUESTION: ... -->` - 明確化が必要
- `<!-- APPROVED: ... -->` - 承認済み
- `<!-- NOTE: ... -->` - 追加情報

#### アノテーション後のプロンプト
```
計画ドキュメントのフィードバックに従って、計画を更新してください。
```

このサイクルを計画が完全に承認されるまで繰り返します。

### 第4段階：実装

計画が確定したら、実装を開始します。

#### プロンプト例
```
docs/exec-plans/active/[task-name]-plan.md に従って、すべてを実装してください。
各タスク完了時に計画ドキュメントで完了をマークしてください。
すべてが完了するまで停止しないでください。
```

#### 実装中の期待事項

エージェントは以下を自動的に実行すべきです：

1. **段階的な実装**
   - 計画の各ステップを順番に実装
   - 各ステップ完了時に計画にマーク（✅）

2. **自己検証**
   - 型チェックを実行
   - リントを実行
   - テストを実行

3. **問題の修正**
   - エラーを検出したら自動的に修正
   - 修正を計画に記録

4. **人間へのエスカレーション**
   - 判断が必要な場合のみ停止
   - 具体的な質問と選択肢を提示

### 第5段階：コミットとPull Request

実装が完了したら、Gitブランチ戦略に従ってコミットし、Pull Requestを作成します。

#### ブランチ戦略

**ブランチ命名規則**:
- `feature/[task-name]` - 新機能の実装
- `fix/[issue-description]` - バグ修正
- `refactor/[component-name]` - リファクタリング
- `docs/[document-name]` - ドキュメント更新
- `chore/[task-name]` - ビルドプロセスや補助ツールの変更

**例**:
```
feature/sensor-data-visualization
fix/temperature-unit-conversion
refactor/api-client-architecture
docs/add-deployment-guide
chore/update-dependencies
```

#### ワークフロー

1. **ブランチ作成**
   ```bash
   git checkout -b feature/[task-name]
   ```

2. **実装とコミット**
   - 計画の各ステップを実装
   - 論理的な単位でコミット（小さすぎず大きすぎず）
   - コミットメッセージは明確かつ説明的に

3. **コミットメッセージの形式**
   ```
   [簡潔なタイトル]（50文字以内）

   [詳細な説明]
   - 何を変更したか
   - なぜ変更したか
   - 影響範囲

   🤖 Generated with [Claude Code](https://claude.com/claude-code)

   Co-Authored-By: Claude <noreply@anthropic.com>
   ```

4. **Push**
   ```bash
   git push -u origin feature/[task-name]
   ```

5. **Pull Request作成**

   PRタイトル:
   ```
   [Type] Brief description of changes
   ```

   PR本文テンプレート:
   ```markdown
   ## 概要
   この変更の目的と背景を簡潔に説明

   ## 変更内容
   - 主要な変更点1
   - 主要な変更点2
   - 主要な変更点3

   ## テスト
   - [ ] 単体テスト追加/更新
   - [ ] 手動テスト完了
   - [ ] リント/型チェック通過

   ## 関連ドキュメント
   - 実行計画: `docs/exec-plans/active/[task-name]-plan.md`
   - リサーチ: `docs/exec-plans/active/[task-name]-research.md`

   ## スクリーンショット（該当する場合）
   [変更前後の画面キャプチャ]

   ## チェックリスト
   - [ ] ARCHITECTURE.mdの原則に従っている
   - [ ] 適切なテストカバレッジ
   - [ ] ドキュメント更新済み
   - [ ] 技術的負債を記録済み（該当する場合）

   🤖 Generated with [Claude Code](https://claude.com/claude-code)
   ```

### 第6段階：PRレビューと修正

**重要**: PRを作成した後も、レビューと修正のプロセスがあります。

#### 自動チェック

1. **CI/CDパイプライン**
   - リント、型チェック、テストが自動実行
   - 失敗した場合は、エラーを確認して修正
   - 同じブランチに修正をコミット・プッシュ

2. **対応が必要なCI失敗例**:
   ```bash
   # 型エラーの場合
   npm run type-check  # ローカルで確認
   # 修正後
   git add . && git commit -m "fix: TypeScript型エラーを修正"
   git push

   # テスト失敗の場合
   npm run test  # ローカルで確認
   # 修正後
   git add . && git commit -m "fix: テストエラーを修正"
   git push
   ```

#### AIレビュアーとの対話

GitHub Appsとして設定されたAIレビュアー（Claude、Codex、Gemini等）からフィードバックが来る場合があります。

**対応手順**:

1. **フィードバックの確認**
   ```bash
   gh pr view [PR番号] --comments
   ```

2. **フィードバックへの対応**
   - AIレビュアーのコメントを読み、提案を確認
   - 必要に応じてコードを修正
   - 同じブランチに修正をコミット

3. **レビュアーへの返信**
   - 修正した内容を説明
   - 対応しない場合は理由を説明
   - 返信の冒頭には必ずレビュアーへのメンションを含める：
     - Gemini → `@gemini-code-assist`
     - Codex → `@codex`
     - GitHub Copilot → `@copilot`

**プロンプト例**:
```
Github Apps に Claude, Codex, Gemini Code Assistant を追加してPRを自動レビューさせるようにしました。
彼らと会話してPRを適切に修正していってください。
```

**エージェントの対応**:
1. PRのレビューコメントを確認
2. 指摘された問題を分析
3. コードを修正
4. コミット・プッシュ
5. 必要に応じてレビュアーに返信

#### 人間のレビュー

1. **コードレビュー**
   - コードの品質とアーキテクチャの整合性を確認
   - 計画との一致を検証
   - フィードバックがあれば、同じブランチで修正

2. **フィードバックへの対応**
   - レビューコメントに基づいて修正
   - 議論が必要な場合はコメントで返信

#### マージ

すべてのチェックとレビューが完了したらマージ:

1. **PR 作成時に自動マージを設定する**
   ```bash
   gh pr merge <PR番号> --auto --squash
   ```

2. **マージ前の最終確認**
   - ✅ AIレビュアー（Gemini・Copilot）のコメントに返信・修正済み
   - ✅ 各 AI レビュアーから OK（「LGTM」等）の返信が来ている
   - ✅ すべての CI チェックが通過
   - ✅ コンフリクトなし

2. **マージ後の作業**
   - 実行計画を `docs/exec-plans/completed/` に移動
   - READMEやNEXT_STEPS.mdを更新（必要に応じて）

#### PRの粒度

**1つのPRにつき1つの論理的な変更**:
- ✅ 良い例: センサーデータのグラフ表示機能を追加
- ✅ 良い例: API認証のバグを修正
- ❌ 悪い例: グラフ表示機能追加 + 認証バグ修正 + リファクタリング

大きな機能の場合は、複数のPRに分割：
```
feature/sensor-dashboard (親)
├── feature/sensor-dashboard-api (PR #1)
├── feature/sensor-dashboard-ui (PR #2)
└── feature/sensor-dashboard-integration (PR #3)
```

### 実装中のフィードバック

人間からのフィードバックは短く、簡潔に：

- ✅ 良い例: 「この関数を実装していません」
- ✅ 良い例: 「UIに2pxのギャップがあります」
- ❌ 悪い例: 「もっといい感じにして」（曖昧）

スクリーンショットや既存コードへの参照を活用します。

## セッション管理

### 長いセッションを推奨

リサーチから実装まで、**単一の長いセッションで実行**することで、
エージェントが継続的に理解を深めます。

### コンテキスト管理

- マークダウン文書は永続的な成果物として保持
- コンテキストウィンドウの圧縮後も残る
- 次回セッションの開始点として機能

## 実行計画のライフサイクル

```
docs/exec-plans/active/[task-name]-research.md
docs/exec-plans/active/[task-name]-plan.md
                ↓ （実装完了後）
docs/exec-plans/completed/[task-name]/
├── research.md
├── plan.md
└── retrospective.md  （振り返り）
```

### 完了後の振り返り

実装完了後、以下を記録します：

```markdown
# Retrospective: [Task Name]

## 何がうまくいったか
- ...

## 何が課題だったか
- ...

## 学んだこと
- ...

## 次回への改善案
- ...
```

## 技術的負債の管理

### 負債の記録

実装中に発見した技術的負債は `docs/exec-plans/tech-debt-tracker.md` に記録：

```markdown
## [日付] - [問題の概要]

**場所**: `src/path/to/file.ts:123`

**問題**:
現在の実装では...

**影響**:
- パフォーマンス: 中
- 保守性: 高
- セキュリティ: 低

**推奨アクション**:
...

**優先度**: 高/中/低
```

### 定期的なクリーンアップ

週次または隔週で「ガベージコレクション」タスクを実行：

```
tech-debt-tracker.md の優先度「高」の項目を1つ選び、
修正計画を作成してください。
```

## 主体性の維持

人間は常に主導権を持ちます：

### 計画段階での決定
- アーキテクチャの選択
- 技術スタックの決定
- トレードオフの評価

### 実装段階での監視
- 提案内容を評価し、受け入れ・修正・スキップを判断
- スコープ管理で追加機能を削除
- 既存インターフェースの保護
- 技術的選択の上書き

## エージェントが苦戦している場合

エージェントが進展できない場合、それはシグナルです：

1. **不足している要素を特定**
   - ツール
   - ガードレール
   - ドキュメント
   - 型定義
   - テストヘルパー

2. **エージェントに修正を記述させる**
   ```
   [問題] を解決するために、[ツール/ドキュメント] を作成してください。
   ```

3. **リポジトリにフィードバック**
   - 新しいツールをコミット
   - ドキュメントを更新
   - テンプレートを追加

4. **次回はスムーズに**
   - 同じ問題は二度と発生しない

## 成功の測定

### 速度の指標
- タスク完了までの時間
- プルリクエストのスループット
- リワーク率

### 品質の指標
- テストカバレッジ
- リントエラー率
- バグ発生率

### プロセスの指標
- 計画の修正回数（減少が良い）
- エスカレーション率（適度が良い）
- ドキュメント鮮度（新鮮が良い）

## まとめ

ハーネスエンジニアリングは、エージェントの実行力を活用しながら、
人間の判断と知識を確実に反映させる方法論です。

**鍵となるのは**:
- 計画と実行の分離
- 永続的なドキュメント
- 機械的な検証
- 継続的なフィードバックループ

このワークフローを忠実に守ることで、高速かつ高品質な開発が実現できます。
